# 1. Responsibility-Driven Design (RDD)

## 1.1 Definition of responsibility

**Responsibility** — the obligations or behaviours of an object in terms of its role in the system.

## 1.2 Doing vs. knowing responsibilities

| Type        | Includes                                                                                                                |
| ----------- | ----------------------------------------------------------------------------------------------------------------------- |
| **Doing**   | Creating an object; doing a calculation; initiating action in other objects; controlling/coordinating activities        |
| **Knowing** | Knowing about private encapsulated data; knowing about related objects; knowing about things it can derive or calculate |
|             |                                                                                                                         |

## 1.3 Role in OO design pipeline

- Problem domain → Domain Model (Week 2) → Design Model (Week 3) → OO Implementation
- RDD guides the transition from domain model to design model by deciding which software class gets which responsibility

> [!info] Image: OO pipeline diagram showing problem domain → domain models → design models → implementation, with GRASP applying at the design model stage — Slide 4

> [!summary] Key takeaways
> - Responsibilities are either doing (action) or knowing (information)
> - RDD is the framework; GRASP is the toolset for applying it
> - Design decisions about responsibility flow from the domain model

# 2. GRASP Overview

## 2.1 Definition of GRASP

**GRASP** — General Responsibility Assignment Software Patterns/Principles. A set of patterns (or principles) for assigning responsibilities to objects. Aids design reasoning in a methodical, rational, and explainable way.

## 2.2 Definition of a pattern

**Pattern** — a named and well-known problem/solution pair that can be applied in new contexts; a recurring successful application of expertise in a particular domain.

## 2.3 Advantages of patterns

- Capture expertise and make it accessible to non-experts in a standard form
- Facilitate easy reuse of successful designs
- Improve understandability
- Facilitate communication via a common vocabulary
- New solutions can be generated by modifying existing patterns

## 2.4 The 9 GRASP patterns

Creator, Information Expert, Low Coupling, High Cohesion, Controller\*, Polymorphism, Indirection, Pure Fabrication, Protected Variations.

> [!note] \*Controller is covered in Workshop 16 supplementary material — not in this lecture.

> [!summary] Key takeaways
> - GRASP gives names to design decisions so they can be reasoned about and communicated
> - Patterns don't invent new solutions — they codify proven ones
> - This lecture covers 5 of the 9: Creator, Information Expert, Low Coupling, High Cohesion (Controller via workshop)

# 3. Creator

## 3.1 Problem statement

Who should be responsible for creating a new instance of class A? _(Doing Responsibility)_
- Object creation is one of the most common OO activities
- Good creator assignment supports Low Coupling and reduces maintenance

## 3.2 Solution criteria (four conditions)

Assign class **B** the responsibility to create instances of class **A** if one or more of these hold (the more the better):

| Condition                                  | Meaning                                |
| ------------------------------------------ | -------------------------------------- |
| B **contains** or compositely aggregates A | Strongest — prefer this                |
| B **records** A                            | B logs or tracks instances of A        |
| B **closely uses** A                       | B has high interaction with A          |
| B **has the initializing data** for A      | B holds the data needed to construct A |

## 3.3 Tiebreaker rule

> [!note] If more than one class qualifies, prefer the class that **contains/compositely aggregates** A. Use the domain model to inspire this relationship (low representational gap).

## 3.4 Contraindications

When creation has significant complexity, delegate to a helper:
- Recycled instances (object pooling for performance)
- Conditionally creating from a family of similar classes based on runtime context

Use a **Concrete Factory** or **Abstract Factory** instead. (Covered in GoF Design Patterns, Weeks 7–9.)

## 3.5 Examples

> [!example] Monopoly — Who creates `Square`?
> - Contains/aggregates `Square` → **Board** ✓
> - Records `Square` → N/A
> - Closely uses `Square` → Board & Piece
> - Has initializing data → N/A
> 
> **Answer: `Board` creates `Square`**
> Reflected in DCD as composite aggregation `Board` ◆──40──> `Square`

> [!info] Image: Monopoly domain model + resulting design sequence diagram showing `create` → `:Board` looping `create` → `:Square` [1 to 40], plus DCD with composite aggregation — Slide 20

> [!example] POS — Who creates `SaleLineItem`?
> `Sale` contains/aggregates many `SaleLineItem` objects → **`Sale`** creates `SaleLineItem`

> [!example] POS — Who creates `Player`?
> `MonopolyGame` has the initializing data for `Player` (player name from user input) → **`MonopolyGame`** creates `Player`

> [!example] Library Borrowing System — Who creates `BorrowedItem`?
> In the domain model, `BorrowingRecord` contains/compositely aggregates `BorrowedItem` (1 to 1..\*)
> → **`BorrowingRecord`** creates `BorrowedItem`
> Operation added: `- createBorrowedItem(): void` on `BorrowingRecord`

> [!info] Image: Library domain model showing `BorrowingRecord` Contains `BorrowedItem` (1 to 1..\*), and incomplete design model with `BorrowedItem` as separate class — Slide 31

> [!summary] Key takeaways
> - Check all four Creator conditions; the one with the most hits (or the one that contains/aggregates) wins
> - Always inspect the domain model first to find the relationship that justifies creation
> - When creation logic is complex, delegate to Factory classes instead

# 4. Information Expert

## 4.1 Problem statement

What is a general principle for assigning responsibilities to objects?
- A design model may have hundreds or thousands of classes and responsibilities
- Given responsibility R (doing or knowing something), who should own R?

## 4.2 Solution principle

**Assign responsibility R to the object that has the information needed to fulfil R.**

Leads to better understandability, maintainability, and extendibility.

## 4.3 Where to look

**Step 1 — Check the Design model first.** If a relevant class already exists there, assign to it.
**Step 2 — Fall back to the Domain model.** Use it to inspire creation of a new design class.

## 4.4 Contraindications

Expert is undesirable when it creates bad dependencies:

> [!example] DB storage anti-pattern (POS)
> - By Expert, `Sale` should store itself to the database
> - But SQL/DB logic is unrelated to application logic
> - Putting DB logic in domain-inspired classes increases representational gap
> - Putting DB logic in every class creates widespread DB technology coupling → poor maintainability
> - **Solution:** delegate to dedicated DB expert helper classes

## 4.5 Examples

> [!example] Monopoly — Who identifies a `Square` by name?
> - Who has information about `Square`? → `Board` aggregates all `Square` objects (from Creator example)
> - **Answer: `Board`** is the Information Expert for `Square`; assign `getSquare(name)` to `Board`

> [!info] Image: Design sequence diagram showing unknown caller sending `s = getSquare(name)` to `:Board` (by Expert), and DCD showing `Board ◆──40──> Square` — Slide 25

> [!example] POS — Who calculates `Subtotal`?
> - Subtotal = quantity × price
> - `SaleLineItem` knows `quantity`; `ProductDescription` knows `price`
> - `SaleLineItem` associates with `ProductDescription`, so it can access `price`
> - **Answer: `SaleLineItem`** calculates Subtotal (`getSubTotal()`)
> - `ProductDescription` is _also_ an Information Expert (it knows `price`), but `SaleLineItem` is preferred — see Low Coupling

> [!example] Library — Who checks if a member reaches the borrowing limit?
> - Need: `borrowLimit` and count of active `BorrowingRecord`
> - `Member` has `borrowLimit` and associates with `BorrowingRecord`
> - **Answer: `Member`** — assign `+ hasReachLimit(): Bool`
> - Note: other designs possible; evaluate alternatives with Low Coupling and High Cohesion

> [!info] Image: Domain model of Library Borrowing System with `Member` (borrowLimit, status, expirationDate) — `Borrows` → `BorrowingRecord` — Contains → `BorrowedItem`; design model showing `Member` with `+ hasReachLimit(): Bool` — Slide 35

> [!summary] Key takeaways
> - Assign responsibility to the class that already holds (or can access) the relevant data
> - Always check the design model first, then fall back to the domain model
> - Expert can conflict with coupling/cohesion goals — always cross-check

# 5. Low Coupling

## 5.1 Definition of coupling

**Coupling** — how strongly one element is connected to, has knowledge of, or relies on other elements.

## 5.2 High vs. low coupling effects

| | High Coupling | Low Coupling |
|---|---|---|
| Dependencies | Many, widespread | Minimal, necessary only |
| Change impact | One change ripples to many classes | Changes localised |
| Understandability | Hard to understand in isolation | Easier to understand |
| Reuse | Hard to reuse | Easier to reuse |

## 5.3 Low Coupling as an evaluative principle

- An _evaluative_ principle — apply it to choose between competing valid designs
- Does not prescribe who gets the responsibility; it helps _eliminate_ worse alternatives
- Minimizes dependencies → more maintainable system and more reusable code

## 5.4 Contraindications

High coupling with **stable and well-understood** elements (e.g., standard Java libraries) is acceptable — propagating changes, understandability, and maintainability are not impacted.

## 5.5 Examples

> [!example] POS — `SaleLineItem` vs `ProductDescription` for `Subtotal`
> Alternative: By Expert, `ProductDescription` could calculate Subtotal.
> Problem: `Sale` would then need to access `ProductDescription` directly to sum all subtotals → **unnecessary coupling** between `Sale` and `ProductDescription`.
> Low Coupling prefers: **`SaleLineItem`** calculates Subtotal (`getSubTotal(qty)`), keeping `Sale` decoupled from `ProductDescription`.

> [!info] Image: DCD showing the bad alternative — `Sale` with dashed dependency arrow to `ProductDescription.getSubTotal(qty)`, labelled "Unnecessary coupling" — Slide 39

> [!example] Monopoly — `Board` vs `Dog` (arbitrary class) for `getSquare`
> Design B: `Dog` calls `Board.getAllSquares()`, then calls `get(name)` on the resulting `Map<Square>`.
> Problems with Design B:
> - `Dog` must connect to `Square` to obtain knowledge → extra coupling
> - If `Square`'s interface changes, both `Dog` and `Board` must change → hard to maintain
> - Extra step to get `Square` → not efficient
> 
> **Design A wins:** `Board.getSquare(name)` — single call, no extra dependency on `Square` from `Dog`

> [!info] Image: Two sequence diagrams — Design A (`:Board` handles `s = getSquare(name)` directly) and Design B (`:Dog` → `:Board` → `sqs:Map<Square>` → `s = get(name):Square`) — Slides 40–41

> [!summary] Key takeaways
> - Low Coupling is always applied alongside other GRASP patterns to evaluate alternatives
> - Avoid connecting a class to another just to route or pass through information
> - High coupling to stable libraries is fine; avoid it with volatile application classes

# 6. High Cohesion

## 6.1 Definition of cohesion

**(Functional) Cohesion** — how strongly related and focused the responsibilities of an element are.

## 6.2 Low cohesion effects

A class with low cohesion has many unrelated responsibilities or does too much work:
- Hard to comprehend
- Hard to reuse
- Hard to maintain
- Leads to bloated classes (e.g., a class with 100 methods × 200 lines = 20,000 lines)

## 6.3 High Cohesion as an evaluative principle

- Assign responsibilities so that each class stays focused on a closely related set of behaviours
- An _evaluative_ principle — use it to choose between competing designs
- High cohesion → easier comprehension, simpler maintainability and enhancement

## 6.4 Relationship to Low Coupling

- High Cohesion and Low Coupling must be **considered together**
- A bloated class tends to have high coupling as well (it needs many external classes to do its many jobs)
- Delegating responsibilities to other objects achieves both higher cohesion _and_ lower coupling

## 6.5 Contraindications

Low cohesion is sometimes necessary to meet strict **non-functional requirements**, e.g., larger less-cohesive classes to reduce transmission/storage overhead for performance.

## 6.6 Examples

> [!example] Monopoly — `MonopolyGame.playGame()` doing everything vs delegating
> If `MonopolyGame` handles all sub-tasks itself (`doA`, `doB`, `doC` all self-called):
> → Class becomes bloated (Poor/Low Cohesion)
> 
> Better design: `MonopolyGame` delegates `doA` to one collaborator, `doB` to another, `doC` to another:
> → Each class stays focused; code is easier to understand and maintain

> [!info] Image: Two sequence diagrams — left shows `:MonopolyGame` calling doA, doB, doC all internally (labelled "Poor (Low) Cohesion"); right shows `:MonopolyGame` delegating to separate objects (labelled "Better") — Slide 47

> [!example] POS — Why does `Sale` (not `Register`) create `Payment`?
> By Expert, `Register` records Payment → `Register` could create `Payment`.
> Problem: In the real-world domain, `Register` is involved in most system operations → `Register` becomes potentially bloated (low cohesion).
> 
> Revised design: `Register` calls `makePayment()` on `Sale`; `Sale` creates `: Payment` internally.
> → `Register` stays focused; `Sale` handles payment-related concerns

> [!info] Image: Two sequence diagrams — original showing `Register` creating `Payment` then `addPayment(p)` to `Sale`; revised showing `Register → Sale.makePayment() → create() → :Payment` — Slide 48

> [!example] Library — Why not assign `hasReachedLimit` to `BorrowingKiosk`?
> `BorrowingKiosk` could technically access `BorrowingRecord` and hence check the limit.
> Problems:
> - Requires a new association between `BorrowingKiosk` and `Member` in the design model → **increases coupling**
> - `BorrowingKiosk` already handles many system-level operations → **increases chance of low cohesion**
> 
> **`Member`** is preferred: it holds `borrowLimit` directly and already associates with `BorrowingRecord`.

> [!summary] Key takeaways
> - Keep each class focused on a single coherent area of responsibility
> - Low Coupling and High Cohesion are inseparable evaluative principles — always apply both
> - Delegation is the key mechanism for achieving both: let each class do its own focused job

# 7. Summary & Interplay

- GRASP guides decisions about who should be responsible for doing or knowing something
- Given the same domain model, many valid designs can satisfy GRASP principles
- **Creator** and **Information Expert** suggest _who_ gets a responsibility
- **Low Coupling** and **High Cohesion** _evaluate_ competing valid suggestions

| Pattern | Type | Core question |
|---|---|---|
| Creator | Assignive | Who creates instances of A? |
| Information Expert | Assignive | Who has the info to fulfil this responsibility? |
| Low Coupling | Evaluative | Does this design introduce unnecessary dependencies? |
| High Cohesion | Evaluative | Does this design bloat a class with unrelated work? |

> [!note] When Creator and Information Expert suggest different classes, use Low Coupling and High Cohesion to decide. Example: `Register` vs `Sale` for `Payment` creation — Expert says Register, but cohesion says delegate to Sale.
